# v2.0.0 — Programmatic TypeScript API

**Codename:** `fluent`
**Breaking:** Yes (major version bump — new exports, restructured internals)
**Estimated scope:** ~2,500 LOC new + ~500 LOC refactored

---

## Motivation

seedforge today is CLI-only. Users who want to seed in test files must shell out to `npx seedforge`. Every serious competitor (Snaplet/Seed, Drizzle Seed) offers a programmatic API. This is the #1 feature that turns seedforge from "a tool you run once" into "a library your test suite depends on."

**Target users:**
- Developers writing integration tests (vitest, jest, mocha)
- CI pipelines that need deterministic test fixtures
- Applications that seed on first boot

---

## Design Principles

1. **Zero-config by default** — `seed(dbUrl)` should just work, no schema file needed
2. **Progressive disclosure** — simple API for simple cases, fluent builder for complex ones
3. **Type-safe where possible** — autocomplete for options, generic table types
4. **Non-breaking CLI** — the CLI continues to work exactly as before, just backed by the new API
5. **Transaction-safe** — support wrapping seeds in a transaction with rollback for test isolation

## Public API Surface

### Tier 1: One-liner (most common)

```typescript
import { seed } from '@otg-dev/seedforge'

// Seed a database with defaults (50 rows/table, all tables)
await seed('postgres://localhost/testdb')

// With options
await seed('postgres://localhost/testdb', {
  count: 10,
  seed: 42,
  exclude: ['audit_log'],
})

// From a schema file (no live DB needed)
await seed({
  prisma: './prisma/schema.prisma',
  output: './seed.sql',
})

// Dry-run: returns generated SQL without executing
const sql = await seed('postgres://localhost/testdb', {
  dryRun: true,
})
```

### Tier 2: Seeder instance (test suites)

```typescript
import { createSeeder } from '@otg-dev/seedforge'

const seeder = await createSeeder('postgres://localhost/testdb', {
  seed: 42,
  transaction: true, // wraps everything in a transaction
})

// Seed specific tables with counts
await seeder.seed('users', 10)
await seeder.seed('posts', 50)

// Seed with overrides (column values / generators)
await seeder.seed('users', 5, {
  columns: {
    role: { values: ['admin'], weights: [1.0] },
    email: { generator: (faker) => faker.internet.email() },
  },
})

// Get generated rows back (for assertions / FK references)
const users = await seeder.seed('users', 3)
// users = [{ id: 1, first_name: 'Marcus', ... }, ...]

const posts = await seeder.seed('posts', 10, {
  columns: {
    author_id: { values: users.map(u => u.id) },
  },
})

// Cleanup: rollback transaction
await seeder.teardown()

// Or: disconnect without rollback
await seeder.disconnect()
```

### Tier 3: Fluent builder (power users)

```typescript
import { createSeeder } from '@otg-dev/seedforge'

const seeder = await createSeeder('postgres://localhost/testdb')

// Chain seeds with automatic FK resolution
const result = await seeder
  .table('users', 5)
  .table('categories', 3)
  .table('posts', 20)       // auto-picks from generated users + categories
  .table('comments', 50)    // auto-picks from generated users + posts
  .execute()

// result.users = [{ id: 1, ... }, ...]
// result.posts = [{ id: 1, author_id: 1, ... }, ...]

await seeder.disconnect()
```

### Tier 4: Test helpers

```typescript
import { withSeed } from '@otg-dev/seedforge'

// vitest / jest integration
describe('user API', () => {
  const { seed, teardown } = withSeed('postgres://localhost/testdb', {
    seed: 42,
    transaction: true,
  })

  afterAll(teardown)

  it('lists users', async () => {
    await seed('users', 10)
    const res = await fetch('/api/users')
    expect(res.json()).toHaveLength(10)
  })
})
```

## Architecture

### New files

```
src/
├── api/
│   ├── index.ts           # Public exports: seed, createSeeder, withSeed
│   ├── seed.ts            # One-liner seed() function
│   ├── seeder.ts          # Seeder class (stateful, per-connection)
│   ├── builder.ts         # Fluent table chain builder
│   ├── helpers.ts         # withSeed() test helper
│   └── types.ts           # SeedOptions, SeederOptions, TableSeedOptions, SeedResult
├── index.ts               # Re-export api/ + existing types
```

### Refactoring required

1. **Extract orchestration from CLI** — `src/cli.ts` currently contains the full pipeline (introspect → generate → execute). Extract into `src/core/pipeline.ts` that both CLI and API call.

2. **Return generated rows** — Currently the generate step produces rows but they're consumed by the executor and discarded. Add a `returnRows` option that captures inserted rows (with DB-generated IDs for auto-increment columns).

3. **Transaction wrapper** — Add transaction support to the direct executors (PG, MySQL, SQLite). The executor should:
   - BEGIN at start
   - INSERT all tables
   - Return a `teardown()` function that calls ROLLBACK
   - Or COMMIT if `transaction: false`

4. **Connection reuse** — The Seeder instance holds a single connection. Multiple `.seed()` calls reuse it. The CLI creates + destroys per invocation (no change).

### Package exports

```jsonc
// package.json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js"
    }
  }
}
```

The existing `import { ... } from '@otg-dev/seedforge'` path currently exports nothing useful. v2.0.0 makes it the primary API surface.

## Implementation Phases

### Phase 1: Core extraction (~400 LOC)

**Goal:** Extract the CLI pipeline into reusable core functions.

1. Create `src/core/pipeline.ts`:
   - `introspectSchema(options)` — connect, introspect, disconnect (or reuse connection)
   - `generateData(schema, options)` — map columns, build graph, generate rows
   - `executeInserts(connection, plan, options)` — insert rows, return results
   - `runPipeline(options)` — full orchestration (introspect → generate → execute)

2. Refactor `src/cli.ts` to call `runPipeline()` instead of inlining the logic.

3. Tests: Ensure all 1133 existing tests still pass. Add unit tests for `pipeline.ts`.

**Files changed:** `src/cli.ts`, new `src/core/pipeline.ts`, new `src/core/types.ts`

### Phase 2: seed() one-liner (~300 LOC)

**Goal:** Ship the simplest API — `await seed(url, options)`.

1. Create `src/api/seed.ts`:
   - Accepts connection string or options object
   - Calls `runPipeline()` with appropriate options
   - Returns `SeedResult` (table names, row counts, generated SQL if dry-run)

2. Create `src/api/types.ts`:
   - `SeedOptions` — mirrors CLI options
   - `SeedResult` — { tables: Map<string, Row[]>, rowCount: number, duration: number }

3. Create `src/api/index.ts` + update `src/index.ts` to re-export.

4. Tests: Integration tests using a local PG database.

**Files:** new `src/api/seed.ts`, `src/api/types.ts`, `src/api/index.ts`, updated `src/index.ts`

### Phase 3: Seeder class (~600 LOC)

**Goal:** Stateful seeder with connection reuse and per-table seeding.

1. Create `src/api/seeder.ts`:
   - `createSeeder(url, options)` — async factory, connects + introspects on creation
   - `seeder.seed(tableName, count, overrides?)` — seed one table, return rows
   - `seeder.teardown()` — rollback transaction + disconnect
   - `seeder.disconnect()` — commit + disconnect

2. **Row return mechanism:**
   - Modify direct executors to optionally return inserted rows
   - For PG: use `INSERT ... RETURNING *`
   - For MySQL: use `SELECT * FROM table ORDER BY id DESC LIMIT N` after insert
   - For SQLite: use `last_insert_rowid()` range

3. **Transaction support:**
   - PG: `BEGIN` / `ROLLBACK` / `COMMIT`
   - MySQL: `START TRANSACTION` / `ROLLBACK` / `COMMIT`
   - SQLite: `BEGIN` / `ROLLBACK` / `COMMIT` (already synchronous)

4. Tests: Full integration tests for seed → assert → teardown cycle.

**Files:** new `src/api/seeder.ts`, modified `src/output/executors/direct.ts`, `mysql-direct.ts`, `sqlite-direct.ts`

### Phase 4: Fluent builder (~400 LOC)

**Goal:** Chainable `.table().table().execute()` API.

1. Create `src/api/builder.ts`:
   - `TableChain` class with `.table(name, count, options?)` method
   - `.execute()` resolves the chain: topological sort, generate in order, insert
   - Returns `Record<string, Row[]>` keyed by table name
   - Auto-resolves FK references between chained tables

2. Wire into `Seeder` class as the fluent alternative.

3. Tests: Multi-table chain with FK resolution assertions.

**Files:** new `src/api/builder.ts`, updated `src/api/seeder.ts`

### Phase 5: Test helpers + docs (~300 LOC)

**Goal:** `withSeed()` helper for vitest/jest, README updates.

1. Create `src/api/helpers.ts`:
   - `withSeed(url, options)` — returns `{ seed, teardown }` for use in test hooks
   - Lazy connection (connects on first `seed()` call)
   - `teardown()` rolls back and disconnects

2. Update README with programmatic API documentation.
3. Add `examples/` directory with vitest and jest examples.

**Files:** new `src/api/helpers.ts`, updated `README.md`

### Phase 6: CLI migration + release (~200 LOC)

**Goal:** Migrate CLI to use the new API internals, version bump, release.

1. Refactor CLI to use `seed()` / `runPipeline()` internally.
2. Bump version to 2.0.0.
3. Update CHANGELOG.
4. Publish.

## Breaking Changes

| Change | Migration |
|--------|-----------|
| `src/index.ts` now exports API functions | Non-breaking — additive |
| Internal module restructure (`src/core/`) | Only affects deep imports (unsupported) |
| Minimum Node version stays 18 | No change |

**The CLI interface is fully backward-compatible.** This is a major version bump for the new API surface, not for CLI breakage.

## Non-goals (for this release)

- Type-safe generated client (like Snaplet's codegen) — too much scope
- Schema migration integration — separate concern
- Watch mode / hot-reload — CLI feature, not API
- Browser/edge runtime support — Node-only for now
