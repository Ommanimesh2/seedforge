# v2.4.0 — Production Data Subset + Anonymization

**Codename:** `mirror`
**Breaking:** No
**Estimated scope:** ~2,000 LOC new

---

## Motivation

The #1 enterprise ask in test data management: "Give me a small, safe copy of production." Tonic.ai charges $50K+/yr for this. Neosync (open-source alternative) is dead. No good open-source Node.js tool exists. This positions seedforge as the complete test data platform — generate synthetic data OR safely mirror real data.

---

## Feature Design

### CLI

```bash
# Subset: copy 1000 users + all their related data from prod to dev
seedforge subset \
  --source postgres://readonly@prod-replica:5432/app \
  --target postgres://localhost:5432/dev \
  --root users \
  --limit 1000 \
  --seed 42

# Subset + anonymize: mask PII during copy
seedforge subset \
  --source postgres://readonly@prod-replica:5432/app \
  --target postgres://localhost:5432/dev \
  --root users \
  --limit 1000 \
  --anonymize

# Export anonymized subset to SQL file
seedforge subset \
  --source postgres://readonly@prod-replica:5432/app \
  --output subset.sql \
  --root users \
  --limit 500 \
  --anonymize

# Anonymize in-place (dev/staging only, never prod)
seedforge anonymize \
  --db postgres://localhost:5432/staging \
  --tables users,customers
```

### Config

```yaml
# .seedforge.yml
subset:
  root: users                    # start table for graph traversal
  limit: 1000                    # max rows from root table
  strategy: breadth-first        # breadth-first | depth-first | random
  include_orphans: false         # include rows with no FK to root?
  max_depth: 5                   # max FK traversal depth

anonymize:
  enabled: true
  rules:
    # Auto-detect PII columns by name pattern (default: true)
    auto_detect: true

    # Column-specific rules
    columns:
      users.email:
        method: faker             # replace with faker data
        generator: internet.email
      users.first_name:
        method: faker
        generator: person.firstName
      users.last_name:
        method: faker
        generator: person.lastName
      users.phone:
        method: mask              # partial mask: 555-***-**89
        visible: 2                # last N chars visible
      users.ssn:
        method: redact            # replace with fixed value
        value: '***-**-****'
      orders.ip_address:
        method: hash              # deterministic hash (preserves uniqueness)
        salt: ${ANON_SALT}
      users.bio:
        method: ai                # LLM-generated replacement (if v2.2.0 AI enabled)
      payments.card_number:
        method: null              # set to NULL

    # Preserve referential integrity
    preserve_fks: true            # anonymized FKs still point to valid rows

    # Deterministic: same input → same output (for reproducible staging envs)
    deterministic: true
    seed: 42
```

### Programmatic API

```typescript
import { subset, anonymize } from '@otg-dev/seedforge'

// Subset
await subset({
  source: 'postgres://prod-replica/app',
  target: 'postgres://localhost/dev',
  root: 'users',
  limit: 1000,
  anonymize: true,
})

// Anonymize in-place
await anonymize('postgres://localhost/staging', {
  tables: ['users', 'customers'],
  rules: {
    'users.email': { method: 'faker', generator: 'internet.email' },
    'users.ssn': { method: 'redact', value: '***-**-****' },
  },
})
```

## Implementation

### Phase 1: Subset extractor (~600 LOC)

**Goal:** Extract a referentially-intact slice of a database.

1. `src/subset/planner.ts`:
   - Takes the schema (tables, FKs) and a root table + limit
   - Builds a directed FK graph from root
   - Plans extraction order (leaf tables first, root last — reverse topo sort)
   - Respects `max_depth` to avoid pulling the entire database
   - Handles circular FKs (break cycles at configurable points)

2. `src/subset/extractor.ts`:
   - Connect to source (read-only)
   - Query root table: `SELECT * FROM users ORDER BY id LIMIT 1000`
   - For each FK from root → child: `SELECT * FROM orders WHERE user_id = ANY($1)`
   - Recurse for each child table's FKs
   - Deduplicate rows (same row reachable via multiple paths)
   - Memory-efficient: stream rows in batches of 1000

3. `src/subset/streamer.ts`:
   - Async generator that yields rows per table
   - Batched queries with cursor-based pagination
   - Connection pooling for parallel table extraction

4. `src/subset/inserter.ts`:
   - Inserts extracted rows into target database
   - Reuses existing `direct.ts` / `sqlite-direct.ts` / `mysql-direct.ts` executors
   - Maintains insertion order (parent before child)
   - Resets sequences after insertion

### Phase 2: Anonymization engine (~500 LOC)

**Goal:** Transform PII columns during copy or in-place.

1. `src/anonymize/detector.ts` — Auto-detect PII columns:
   ```typescript
   const PII_PATTERNS = {
     email: /email|e_mail/i,
     phone: /phone|mobile|cell|fax|tel/i,
     name: /first_name|last_name|full_name|display_name/i,
     address: /address|street|city|state|zip|postal/i,
     ssn: /ssn|social_security|national_id|tax_id/i,
     credit_card: /card_number|cc_number|credit_card/i,
     ip: /ip_address|ip_addr|remote_ip/i,
     dob: /birth_date|date_of_birth|dob/i,
     password: /password|passwd|pwd/i,
   }
   ```

2. `src/anonymize/methods/`:
   - `faker.ts` — Replace with faker value (seeded for determinism)
   - `mask.ts` — `"555-1234"` → `"***-**34"` (configurable visible chars)
   - `redact.ts` — Replace with fixed string
   - `hash.ts` — `SHA256(salt + value).hex().slice(0, originalLength)` — preserves uniqueness and length
   - `null.ts` — Set to NULL

3. `src/anonymize/transformer.ts`:
   - Takes a row and a set of rules
   - Returns a new row with anonymized columns
   - Preserves non-PII columns exactly
   - Preserves FK values (anonymized emails still point to valid user IDs)

4. `src/anonymize/rules.ts`:
   - Resolves config rules to method functions
   - Merges auto-detected rules with explicit overrides
   - Explicit rules always win over auto-detect

### Phase 3: CLI subcommands (~300 LOC)

1. Add `subset` subcommand to CLI:
   ```
   seedforge subset --source <url> --target <url> --root <table> --limit <n>
   ```

2. Add `anonymize` subcommand:
   ```
   seedforge anonymize --db <url> --tables <t1,t2>
   ```

3. Safety checks:
   - **NEVER anonymize production** — detect prod URLs (RDS, Cloud SQL, Neon, Supabase) and block
   - Require `--yes --i-know-what-im-doing` flag for in-place anonymization
   - Source connection should be read-only when subsetting (warn if not)
   - Show row count estimates before starting

### Phase 4: Integration + testing (~300 LOC)

1. Wire into programmatic API: `subset()`, `anonymize()` exports.
2. Integration tests with local PG database:
   - Subset: create source DB, seed it, subset to target, verify referential integrity
   - Anonymize: seed DB, anonymize, verify PII columns changed, FKs intact
3. E2e tests for CLI subcommands.

## Safety Model

This feature touches production data. Safety is non-negotiable.

| Guard | Description |
|-------|-------------|
| **Read-only source** | Subset never writes to source. Warn if source user has write perms. |
| **Prod detection** | Block anonymize on prod-like URLs. Require double-opt-in. |
| **Dry-run** | `--dry-run` shows what would be extracted/anonymized without doing it. |
| **Row limits** | Default max 10,000 rows from root. Must explicitly increase. |
| **No password columns** | Auto-detect `password` columns, always redact (never copy hashes). |
| **Audit log** | Log what was anonymized (column names, not values) to stdout. |

## Success Criteria

- [ ] `seedforge subset --source A --target B --root users --limit 100` works
- [ ] Extracted data is referentially intact (all FKs resolve)
- [ ] Circular FKs handled (cycle-breaking with configurable strategy)
- [ ] `--anonymize` flag masks PII columns during copy
- [ ] Auto-detect finds email, phone, name, SSN, address, card_number columns
- [ ] 5 anonymization methods work: faker, mask, redact, hash, null
- [ ] Deterministic anonymization (same input + seed → same output)
- [ ] `seedforge anonymize --db` works for in-place anonymization
- [ ] Production safety guards block dangerous operations
- [ ] Works with PG, MySQL, SQLite
- [ ] Programmatic API: `subset()`, `anonymize()` exported
- [ ] ≥40 new tests
- [ ] Streaming extraction (memory-flat for large datasets)

---

## Non-goals

- MongoDB or NoSQL support
- Schema migration during subset (source and target must have same schema)
- Incremental/delta subsetting (always full extraction)
- Data generation to fill gaps after subsetting
- GDPR compliance certification (we provide the tools, not the legal guarantee)
