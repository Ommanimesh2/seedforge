# v2.1.0 — Weighted Relationship Cardinality

**Codename:** `cardinal`
**Breaking:** No
**Estimated scope:** ~800 LOC new

---

## Motivation

Currently, seedforge generates a flat N rows per table. Real data has variable cardinality — some users have 50 orders, most have 2-3, some have 0. Drizzle Seed has this feature and it's a common ask. This makes generated data dramatically more realistic.

---

## Feature Design

### Config syntax

```yaml
# .seedforge.yml
tables:
  users:
    count: 20

  orders:
    count: 100
    relationships:
      user_id:
        cardinality: 1..10        # each user gets 1-10 orders
        distribution: zipf        # most users have few orders, some have many
        # OR:
        cardinality: [1, 3, 5]    # exact values to pick from
        weights: [0.6, 0.3, 0.1] # weighted

  order_items:
    count: 300
    relationships:
      order_id:
        cardinality: 1..5         # each order has 1-5 items
        distribution: uniform

  reviews:
    relationships:
      user_id:
        cardinality: 0..3         # 0 means some users have no reviews
        nullable_rate: 0.2        # 20% of users get 0 reviews
```

### Programmatic API

```typescript
const seeder = await createSeeder(dbUrl)

const users = await seeder.seed('users', 20)
const orders = await seeder.seed('orders', {
  relationship: {
    user_id: {
      source: users,
      cardinality: { min: 1, max: 10, distribution: 'zipf' },
    },
  },
})
```

## Implementation

### Phase 1: Cardinality resolver (~300 LOC)

Create `src/generate/cardinality.ts`:

1. `CardinalitySpec` type:
   ```typescript
   interface CardinalitySpec {
     min: number
     max: number
     distribution: 'uniform' | 'zipf' | 'normal'
     weights?: number[]  // for discrete cardinality values
   }
   ```

2. `resolveCardinality(spec, faker)` — returns a number (how many child rows for this parent)

3. `distributeRows(parentIds, totalChildCount, spec, faker)`:
   - Takes parent IDs and total desired child rows
   - Returns a `Map<parentId, count>` distributing children across parents
   - Respects min/max bounds
   - Uses the specified distribution

### Phase 2: Integration with generator (~300 LOC)

Modify `src/generate/generate-table.ts`:

1. Before generating rows for a table, check if any FK column has a `relationships` config.
2. If yes, call `distributeRows()` to pre-assign how many rows each parent gets.
3. During row generation, assign FK values according to the distribution plan instead of random selection from the reference pool.

### Phase 3: Config + validation (~200 LOC)

1. Add `relationships` to the config schema in `src/config/types.ts`.
2. Validate in `src/config/validate.ts`:
   - `cardinality` must be a range string ("1..10"), array, or object
   - `min` ≤ `max`
   - `weights` length must match array cardinality
   - Referenced column must be a FK column

3. Parse range strings in `src/config/loader.ts`.

## Success Criteria

- [ ] Config-based cardinality: `cardinality: 1..10` works
- [ ] Distribution support: zipf, uniform, normal
- [ ] Weighted discrete values: `cardinality: [1, 3, 5]` with weights
- [ ] Zero cardinality (some parents have no children)
- [ ] Works with programmatic API `seeder.seed()` overrides
- [ ] Total child row count still respects `count` config
- [ ] Existing tests pass (backward compatible — default is current behavior)
- [ ] ≥20 new tests

---

## Non-goals

- Cross-table correlated cardinality (e.g., "premium users have more orders") — future work
- Cardinality for composite FKs — v2 only supports single-column FK cardinality
